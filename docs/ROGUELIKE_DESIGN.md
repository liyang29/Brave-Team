## 0. 方向转变（为什么不是原来那个游戏了）

原项目是"公会经营 + 派遣 + 自动战斗"的模拟经营。白盒试玩后判定**不好玩**，诊断出根因：经营/旁观双层都浅，是"有新鲜感无深度"的开罗游戏式循环（详见旧 DEVELOPMENT 讨论与记忆）。

结论：**深度最容易在"战斗/构筑"层做出**。于是转向——

> **一个以"背包构筑 + 小队 + roguelike 旅程"为核心的游戏。**

参照 Megaloot / Backpack Hero / Backpack Battles（这些都是**单角色**背包游戏），**我们的差异化 = 小队**（3~5 人，每人独立背包 + 站位）。

## 1. 一句话定位与核心循环

> 带一支小队（村长视角），边走边搭每个人的背包，打到魔王城。深度在**搭背包**，战斗**自动结算**。

```
组建/调整小队背包（搭物品+技能书+副属性+站位）
      ↓
遭遇 → 自动战斗（回合制结算，快揭晓）
      ↓
拿战利品 / 商店 / 选路线
      ↓
下一关（地形/威胁变化）……直到魔王城 / 团灭
      ↓
（局间）meta 解锁 → 下一局
```

## 2. 核心支柱（已在实验中验证/实现）

### 2.1 背包构筑（核心深度）
- 每人一个**网格背包**，物品占格 → 空间有限 = 取舍。
- **邻接协同**：相邻物品触发加成（开刃/重装/共鸣/生机）→ 摆位 = 深度。
- 力量**全来自背包，不练级**（背包就是成长曲线）。
- 实现：`scripts/experiments/BackpackModel.gd`（纯数据/函数）。

### 2.2 技能书（技能也是背包物品）
- 技能 = 占格的**技能书**，和装备**抢空间**（带书=少带甲）。
- **认职业**（火球书只法师能用…）。
- **回合冷却**（带很多书也不能一回合倾泻；冷却挂书上、经 Party 注入）。

### 2.3 站位（世界树 Etrian Odyssey 式"软调整"）
- 前/后排（实验为前3/后3网格，仅前后排计入战斗，列只是槽位）。
- **软模型**（`Party.positioning_mode = "soft_row"`）：人人可被打到，站位只改伤害——
  - 后排发起近战(物理) ×0.5；后排承受物理伤 ×0.7（可叠乘）；魔法/远程不受影响。
  - 前排至少留 1 人；前排全灭后排自动顶上。
- 另有"硬触及"模式（`"reach"`，默认）供别的实验用，互不影响。
- 实现：`BattleSimulator._row_damage_mult` / `_promote_if_front_empty`。

### 2.4 副属性（可扩展地基 + 暴击）
- `BattleCombatant.extra_stats` 字典 + `get_stat()`，经 `Party.extra_stats_config` 注入。
- 加新副属性（法抗/吸血/破甲…）≈ 改数据：扩 `BackpackModel.EXTRA_KEYS` + 物品声明 key + 公式读 `get_stat()`。
- 已实现：**暴击**（crit_chance 概率 + crit_dmg；倍率=1.5+crit_dmg；普攻/单体/AOE 都吃）。
- 完整属性菜单见 `COMBAT_STATS_CATALOG.md`。

### 2.5 战斗模型
- **回合制 + 自动结算**（复用 `BattleSimulator`）。**否决实时制**（会让谜面模糊、需重写）。
- 想要"实时观感"靠后期动画播放回合结果，不做真实时。

## 3. 当前原型现状（白盒，全在 main 的 `scenes/experiments/`）

| 场景 | 验证了什么 | 结果 |
|------|-----------|------|
| `PositionExperiment.tscn` | 2 排站位（硬触及）| 同队换配队即结果反转 |

- [ ] 真实派遣外壳替换为"商队远征"框架

### 预留到后期（已讨论、暂不做）
- [ ] **AI 选目标考虑站位**（治本）：`choose_target` 按"有效伤害=基础×站位倍率"选目标，避免前排去捅打折的后排等"瞎打"。
- [ ] **玩家目标优先级策略按钮**（安全阀）：集火残血 / 点后排威胁 / 打高攻 / 各自本能（战前设、自动执行）。先治本 AI 再视情加。注意保持"轻"，别做成 FF12 Gambit。
- [ ] 公共驮兽（马/骆驼）= 共享后勤背包（战斗中不可掏，要有代价）。
- [ ] 负重、金币经济、更多副属性（法抗/吸血/破甲…见 stats 目录）、技能书参与邻接协同。
- [ ] 站位"前→后排"数值是否改为 ×0.5（更还原世界树，当前 ×0.7）——待玩后定。

## 5. 工程结构建议（迁移而非重建）

- **战斗/背包核心可整块搬走**（与经营 Manager 基本无耦合，唯一小耦合是 `Party.from_dict` 引用 HeroManager）。
- **建议**：核心玩法验证到位后，**开新项目**把战斗核心搬过去、按 roguelike 架构从头搭骨架；旧经营层（GuildManager/FacilityManager/QuestManager/MapManager/TurnManager/WorldMap/HeroAI 等）留在旧项目当存档。
- **现在**：继续在当前仓库的 `scenes/experiments/` 里验证乐趣，**不急着重建项目**。

## 6. 关键设计决策（含为什么）

| 决策 | 结论 | 为什么 |
|------|------|--------|
| 时间模型 | **回合制 + 自动结算** | 可读、可测、好平衡；实时会糊谜面 + 重写 |
| 深度核心 | **背包构筑**（不是战斗操作）| 只做深一个系统；战斗轻、深度压背包（Megaloot 验证可行）|
| 站位 | **世界树式软调整**（soft_row）| 靠百分比引导分工、更顺滑容错；硬触及留作备选 |
| 力量来源 | **装备/背包，不练级** | 背包即成长曲线；"装备↔职业/任务不兼容"是好的取舍来源 |
| 技能 | **技能书（占格+认职业+回合CD）** | 把技能并入背包这一个深系统，制造"技能 vs 护甲"取舍 |
| 副属性 | **extra_stats 字典地基** | 以后加 A 档纯数字属性≈改数据，不动结构 |
| 差异化 | **小队**（vs 单人背包大作）| 同类都单角色，小队是护城河 |
