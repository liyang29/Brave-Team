# 内容扩展路线 & 工具策略

> 后期要加很多怪物 / Boss / 节点类型 / 地图时，怎么高效编辑与测试。
> 核心结论：**大概率不需要自建编辑器**；投资在"数据表 + 策略模式 + 生成器 + 自动 harness + Debug 菜单"。

## 0. 关键区分：数据 vs 行为

| 类型 | 例子 | 扩展方式 |
|------|------|---------|
| **纯数据** | 怪物数值、物品、节点布局、地图、掉落权重、价格 | **加数据表条目**（无脑加一行） |
| **行为/逻辑** | Boss 特殊机制、独特 AI、节点类型效果 | **加代码，但用注册表/策略模式让加起来很便宜** |

90% 内容是数据；只有"新机制"才需写代码。把两者分开，扩展不痛。

## 1. 要不要自建编辑器？——现阶段不要，可能很久都不要

自建编辑器是几周的大坑，只在"非程序员频繁录内容"时才值。单人 + 自己写代码 + 有 AI 协助 → 负投资。

- **真想可视化调数值时的免费编辑器**：把数据做成 Godot `Resource`(`.tres`)，自带 Inspector 即现成编辑器。代价：文件多加载略慢（见 METHODOLOGY「数据表用大 const dict 别拆一堆 .tres」那条——那条针对性能，可视化编辑是另一个合理理由；几百量级无压力）。在那之前**纯数据表/JSON 更适合**（可 diff、可版本管理、AI 可批量改）。

## 2. 各类内容的正确扩展方式

| 内容 | 做法 | 现状 |
|------|------|------|
| **怪物（很多种）** | `MonsterFactory` + `ENEMIES` 表，加怪=加一行 | ✅ **已做**（`scripts/systems/factories/MonsterFactory.gd`） |
| **Boss 玩法逻辑** | 策略模式：新 Boss = 新 `CombatStrategy` 子类 + 表里一行 ai_type，核心战斗不动 | 骨架已有（`CombatStrategy` + `EnemyAIFactory`），顺着加 |
| **节点类型（柴火/休息/事件/精英…）** | "节点类型注册表"：type → 场景/处理器。现 `RunMap`/`RunManager` 的 `match type` 涨到十几种时升级为注册表 | rest/shop 已是节点类型，分支用 match（待升级注册表） |
| **地图（森林等，十几格）** | **程序生成**，别手摆：写地图生成器吃"生物群系配置"（允许节点类型+权重+长度+分支规则）→ 产出节点图 | ⬜ 未做（现为手写线性 `_build_map`） |
| **物品** | 大 const 表（已是） | ✅ `BackpackModel.ITEMS` + `LootTable` |
| **职业/英雄** | `HeroFactory` 数据驱动（已是） | ✅ |

## 3. 测试 & 调试工具（高 ROI，比编辑器重要）

1. **自动模拟 harness**（已有 `test_balance.gd`）：内容一多无法手动试遍，靠 harness 跑"好/中庸/烂 build × N 局"出胜率，自动发现 Boss 过强 / 物品过劳 / 无敌组合。**扩展期测试命根子。**
2. **游戏内 Debug 菜单**（⬜ 待做，很便宜很值）：开发用作弊面板——跳到任意节点、塞任意背包、打任意 Boss。手动试某 Boss 手感时免从头打。
3. **数据文件即"文本记载工具"**：const 表 / 单个大 JSON，可 diff、可版本管理、AI 可批量生成。

## 4. 分期落地（别过度工程）

- **现在**：const 表 + 策略模式，工具先别做。
- **怪物变多** → ✅ `MonsterFactory`（已做）。
- **想做多张/更大地图** → 写**地图生成器**（吃群系配置）替代手摆。← 下一个该投资的
- **Boss 玩法变多** → 照 `CombatStrategy` 加策略类 + 注册。
- **节点类型变多** → 把 `match type` 升级为节点类型注册表。
- **手动调试变烦** → 加 Debug 跳关/造 build 菜单。
- **真要可视化录数据 / 拉策划进来** → 才考虑 Resource(.tres)+Inspector，最后才自建编辑器。

> 一句话：已有对的骨架（工厂+策略+数据表）。扩展 = 补全骨架 + 加生成器 + 加 harness/Debug 菜单，不是做编辑器。
